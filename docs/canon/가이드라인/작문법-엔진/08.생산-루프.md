# ⚙️ 08. 생산 루프 — 48,000화를 버티는 하네스

> 00.엔진-총론의 5단 순환(①결말 역산 ②복선 장부 ③서브액트 ④화 집필 ⑤검증)을
> **반복 가능한 기계 구조**로 만든 것이 이 문서다. 파일럿(1~3화)이 루프 본체의 1회전 실증이다.

## 1. 층위별 루프 주기 — 언제 무엇을 재실행하는가

| 경계 | 재실행 | 담당 | 산출물 |
|---|---|---|---|
| **화** (매회) | ④집필 → ⑤게이트 | sonnet + 기계 | `원고/…/NNN화.md` |
| **장** (~100화) | ③콘티 재생성 + 표본 패널 | opus | `ActN-부-장-화별콘티.md` |
| **부** (300화) | ②장부 결산(설치/회수 대조) + 설계 재확인 | opus + 기계 | 장부 상태 갱신 |
| **Act** (1,200화) | ①결말 역산 재실행 + 차기 Act 설계 | opus 패널 | 차기 서브액트 설계안 |
| **크로니클** | 명제·로그라인 재검(작가) | 작가 + 오케스트라 | 로드맵 개정 |

핵심: **화 루프는 ④⑤만 돈다.** 매화 설계를 다시 하지 않는다 — 콘티가 소진될 때(장 경계)만 위층으로 올라간다.

## 2. 화 단위 루프 본체 (1사이클)

```
[콘티 확인] 화별콘티에서 해당 화 행 로드 (장면 공식 + FS-ID + 클리프행어 유형)
     ↓
[집필] sonnet — 엔진 01~07 계약 + 스토리-상태 장부의 직전 시놉시스 주입
     ↓
[게이트 1 · 기계] python tools/validate_manuscript.py  ← 분량·금지어·시스템창·클리프행어
     ↓ (FAIL → 같은 에이전트 1회 수정 → 재게이트, 2연속 FAIL → 오케스트라 심사)
[게이트 2 · 캐논] python tools/validate_canon.py  ← 원고가 캐논 링크를 깨지 않는가
     ↓
[상태 갱신] 스토리-상태 장부에 시놉시스 3문장·인물 위치·복선 이벤트 추가 (기계 append)
     ↓
[체크포인트] 5~10화 묶음 커밋·푸시  ← 한도 사고 시 git status 실측 → 잔여 재기동 (인수인계 §3 동일)
```

## 3. 연속성의 단일 진실원 — 스토리-상태 장부

`원고/스토리-상태.md`가 **집필 에이전트가 읽는 유일한 "지금까지의 이야기"**다.
에이전트에게 이전 원고 전문을 읽히지 않는다(컨텍스트 폭발 + 문체 오염). 장부만 읽는다.

스키마(화당 1블록, 기계 append):
- `화번호 | 시놉시스 3문장 | 에반 위치/상태 | 등장 인물과 관계 변화 | 복선 이벤트(FS-ID 설치/강화/회수) | 클리프행어(다음 화 첫 장면이 받아야 할 것)`

장 경계마다 오케스트라가 장부를 **1문단 리캡으로 압축**(원본 블록은 보존) — 장부 자체가 비대해지는 것을 막는다.

## 4. 품질 드리프트 방지 — 2단 검증 구조

| 층 | 주기 | 무엇 | 비용 |
|---|---|---|---|
| 기계 게이트 | 매화 | validate_manuscript + validate_canon | ~0 |
| 표본 패널 | 장 경계(=~100화)마다 + 무작위 5% | 톤 12렌즈 축소판(3렌즈: 문체 일관·캐논 정합·보상 곡선) | opus 소수 |

교훈 반영: V1 사고(어휘 정화가 톤을 뒤집음)처럼 **루프가 오래 돌수록 드리프트는 반드시 생긴다.**
표본 패널이 드리프트를 잡으면 → 원인 화 구간 특정 → 해당 구간만 재집필 배치. 전면 재작업 금지.

## 5. 복선 장부 자동 대조 (부 경계 결산)

06.복선-장부의 각 FS 엔트리에는 설치/강화/회수 예정 좌표가 있다. 부 경계마다 기계 스캔:
- **설치 예정인데 해당 화 원고에 흔적 없음** → 경고 (다음 장 콘티에 재배선)
- **회수 화가 지났는데 상태가 '설치'** → 위반 (오케스트라 심사)
- 회수 완료 → 상태 '회수 완료'로 갱신 + 원고 화번호 기록

## 6. 오케스트레이션 분업 (인수인계 §0과 동일 원칙)

- **판정과 적용 분리**: 콘티·판정은 opus JSON 출력 → 원고 파일 조작·장부 append는 기계/집필 에이전트
- **검수 불신**: 검증관의 파괴적 주장(연속성 붕괴 등)은 장부·원고 대조 후에만 수용
- **한도 사고**: 원고는 파일로 남는다 — git 실측 → 체크포인트 → 스토리-상태 장부 기준 재개
- **작가 게이트**: 장 경계 산출물(콘티)과 부 경계 결산은 작가 검토 큐(인수인계 §7-2)에 등재. 화 단위는 초고 상태로 누적, 작가는 묶음으로 검토

## 7. 착수 순서 (현재 위치)

1. ✅ Act 1 설계안 (레퍼런스) → 2. ✅ 1장 화별 콘티 + 1~3화 파일럿 = **루프 1회전 실증** →
3. ⬅️ **지금**: 하네스 정본화(이 문서 + validate_manuscript.py + 스토리-상태 장부 개통) →
4. 작가 승인 후: 4~100화 화 루프 가동 (5~10화 체크포인트) → 5. 장 경계에서 2장 콘티 → … →
6. 1-B 진입 전 Act 2 설계 병행 착수 (전체 줄거리를 미리 다 잡지 않는다 — Act 경계 재실행 원칙)
